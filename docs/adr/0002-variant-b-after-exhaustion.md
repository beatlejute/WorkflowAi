# ADR 0002: Вариант B (gate после исчерпания) вместо варианта A (блокирующий немедленно)

## Статус

Принято

## Контекст

При реализации human-gate для workflow-ai 1.3.0 рассматривались два варианта:
- **Вариант A**: Блокирующий немедленно — при созревании human-тикета pipeline останавливается сразу
- **Вариант B**: Gate после исчерпания — pipeline добивает non-human eligible тикеты, затем встаёт на gate

Реализационная сложность вариантов:
- Вариант A: требует немедленной остановки pipeline, сложная логика приоритетов
- Вариант B: двойной проход в `pick-next-task.js`, существующая инфраструктура `manual-gate`

## Решение

Выбран вариант B (gate после исчерпания non-human eligible). Причины:
- Решение пользователя: «дать пайплайну добить остальное, не блокировать преждевременно»
- Реализация проще: `pick-next-task.js` делает двойной проход, runner логика не меняется
- Минимальные изменения в существующем коде
- Сохраняется предсказуемость workflow

## Последствия

1. **Плюсы**:
   - Pipeline продолжает обработку non-human задач без остановки
   - Реализация проще и менее подвержена ошибкам
   - Сохраняется существующая инфраструктура `manual-gate`

2. **Минусы**:
   - Human-тикет может ждать дольше, если много non-human задач
   - Требуется careful документация поведения

3. **Митигация**:
   - Conditions у human-тикетов обычно требуют завершения большинства non-human предков
   - VS Code extension (PLAN-025) показывает `pending_count` для информирования пользователя

Trade-off: задержка human-тикета vs предсказуемость и простота реализации. Выбран баланс в сторону предсказуемости.
