# ADR 0003: `mark-blocked` как отдельный скриптовый стейдж, а не модификация `executeUpdateCounter`

## Статус

Принято

## Контекст

При реализации автоблокировки тикетов после исчерпания max_review_attempts рассматривались варианты:
1. Модификация существующего `executeUpdateCounter` (runner.mjs:1603-1620)
2. Создание отдельного скриптового стейжа `mark-blocked`

`executeUpdateCounter` — синхронная функция, которая:
- Принимает только counter и max значения
- Не знает о тикетах и их frontmatter
- Не имеет доступа к файловой системе для записи alerts.jsonl

## Решение

Выбран вариант создания отдельного скриптового стейжа `mark-blocked`. Причины:

1. **Принцип единственной ответственности (SRP)**:
   - `executeUpdateCounter` должен считать попытки, а не записывать frontmatter
   - `mark-blocked` занимается нотификацией и enriching тикета

2. **Тестируемость**:
   - Отдельный скрипт легко unit-тестировать
   - Можно мокать файловые операции и проверки

3. **Переиспользуемость**:
   - Аналогичная логика может использоваться для других причин блокировки
   - Параметризация через `--reason` позволяет расширять функциональность

4. **Pipeline-конфигурация**:
   - Стейдж явно описан в pipeline.yaml
   - Легко добавить дополнительные условия или goto-логику

## Последствия

1. **Архитектурные преимущества**:
   - Чёткое разделение ответственности
   - Pipeline декларативно описывает граф выполнения
   - Скрипт может использоваться в других контекстах

2. **Реализационные**:
   - Требуется новый агент `script-mark-blocked` в pipeline.yaml
   - Стейдж вставляется между `increment-task-attempts.max_reached` и `move-ticket`
   - Скрипт принимает параметры через pipeline context

3. **Обратная совместимость**:
   - `executeUpdateCounter` остаётся без изменений
   - Новая функциональность добавляется без слома существующей

Этот подход обеспечивает чистую архитектуру и соответствует принципам pipeline-as-code.
