# Воркфлоу: Выполнение задачи

Единый универсальный воркфлоу для всех типов задач. Подход определяется содержимым тикета (описание + DoD), а не его типом.

## Вход

- Тикет в `in-progress/` (или `review/` при повторном выполнении)
- Извлечены: описание, DoD, `context.files`, `context.notes`

## Шаги

### 1. Проверить существующий прогресс

**ОБЯЗАТЕЛЬНО перед началом работы** проверь, не выполнялся ли тикет ранее:

1. **Секция `## Result` / `## Результат выполнения`** — если есть, тикет уже частично или полностью выполнен
2. **Секция `## Ревью`** — если есть, тикет проходил ревью; прочитай замечания ревьюера
3. **Пункты DoD** — `[x]` = уже выполнено и подтверждено, `[ ]` = ещё не выполнено

**Правила продолжения:**

| Ситуация | Действие |
|----------|----------|
| Result есть, ревью нет | Проверь незавершённые пункты DoD (`[ ]`). Доделай только их. Дополни существующий Result, не переписывай |
| Result есть, ревью без замечаний к качеству | Доделай только невыполненные пункты DoD. Дополни Result |
| Result есть, ревью с замечаниями | Исправь **только** то, что указано в замечаниях ревью. Остальное не трогай |
| Result нет | Выполняй тикет с нуля (шаги 2-5) |

> Детальная семантика полей тикета: `knowledge/ticket-structure.md`

### 2. Анализ контекста и планирование

1. Прочитать все файлы из `context.files` инструментом Read
2. Изучить `context.references` — внешние ссылки, документация
3. Прочитать `context.notes` — дополнительный контекст от создателя тикета
4. Если тикет ссылается на план (`parent_plan`) — прочитать план для понимания общей картины
5. Определить из описания и DoD, что конкретно нужно сделать (или что **осталось** — с учётом шага 1)
6. Составить мысленный план: какие файлы менять/создавать/проверять, какие инструменты, зависимости и риски

> Если задача с > 5 шагами DoD или continuation сессии — загрузи `knowledge/context-checkpoints.md`

### 3. Выполнение

Действовать по описанию и DoD тикета. Подход определяется содержимым, а не типом:

- **Код:** следовать методологиям TDD, SOLID, DRY; если есть тесты — запустить после изменений
- **⚠️ Тесты на маппинг/парсинг данных:** если тесты проверяют функцию, которая обрабатывает данные из runtime (логи, конфиги, API-ответы, события UI), **обязательно** прочитай реальный источник данных (файл лога, конфиг, пример API-ответа) и используй в fixtures **реальные значения** оттуда. Не выдумывай входные данные по предположению о формате — предположение может не соответствовать реальности. Проследи цепочку данных от источника до функции: какое поле приходит в какой параметр, с каким значением
- **Файлы/конфигурация:** обязательный цикл Read → Edit/Write → Verify (перечитать для проверки)
- **Тестирование:** выполнить чеклист проверок из DoD, зафиксировать pass/fail по каждому пункту
- **Исследование:** использовать доступные инструменты для сбора данных, подкреплять источниками
- **Документация:** проверить актуальность, использовать единый стиль проекта

> Алгоритм выбора подхода и чеклист выполнения: `algorithms/execution-strategy.md`

#### ⛔ Инкрементальная запись (ОБЯЗАТЕЛЬНО)

После выполнения **каждого пункта** DoD — **сразу** запиши результат в тикет:

1. Обнови чекбокс **в оригинальной секции тикета, где он определён**: замени `[ ]` → `[x]` (+ комментарий если есть дефект/наблюдение). Это относится ко **всем** чеклистам тикета — основной чеклист проверок, DoD, и любые другие секции с `[ ]`.
2. Добавь/дополни описание результата в секции Result (что сделано, evidence)

⛔ **НЕ создавай дубль-чеклист в секции Result.** Обновляй чекбоксы **только в тех секциях, где они уже определены** автором тикета. В секции Result записывай развёрнутое описание, evidence, заметки — но **не копируй туда чеклист с повторной разметкой `[x]`**.

⛔ **НЕ дублируй чеклист в stdout-ответе.** В stdout выводи **одно-двухстрочное содержательное summary** без перечисления пунктов DoD с маркерами `[x]`, `✅`, `✓`.

Это защищает от потери прогресса при обрыве сессии. Не откладывай запись на конец — записывай по ходу.

### 4. Верификация по DoD

Для каждого критерия из Definition of Done:

1. Проверить выполнение
2. Если критерий не выполнен — доделать
3. Если критерий невыполним — зафиксировать причину в заметках

> Таблица соразмерности проверки критерию и правило обязательной верификации запуском: `algorithms/execution-strategy.md` разделы «Соразмерность проверки критерию» и «Правило обязательной верификации запуском»

#### ⛔ Соразмерность проверки критерию

Перед тем как считать критерий выполненным, сверь **тип свойства, которое описывает критерий**, с **типом свойства, которое реально проверяет твой метод верификации**. Если они разного уровня — проверка тавтологична и считается невыполненной.

**Таксономия свойств, которые может описывать критерий:**

| Тип свойства в критерии | Ключевые признаки в формулировке | Минимально достаточный способ проверки |
|---|---|---|
| **Структурное** (наличие/существование объекта) | «есть», «присутствует», «создан», «добавлен», «содержит поле» | Проверка существования объекта (assertion на наличие узла, файла, ключа, поля) |
| **Количественное** | число, порог, «не менее», «не более», «ровно N», процент | Измерение значения и сравнение с порогом |
| **Визуальное** (как объект выглядит пользователю) | «оформлен как», «выглядит», «читаем», «контрастен», «виден», «выровнен», «не обрезан», «не голый», «стилизован», «overflow», запреты на внешний вид | **Недостаточно** проверки существования узла. Требуется либо (а) измерение визуального атрибута из того же рендер-контекста, в котором работает конечный пользователь (например, вычисленные стили, геометрия элемента, цвет пикселей), либо (б) визуальный артефакт (снимок) с явной ручной или мультимодальной сверкой содержимого против формулировки критерия |
| **Поведенческое** | «при клике», «после отправки», «в ответ на», «переключается», «обновляется» | Выполнить действие и проверить наблюдаемый результат в том же рендер-контексте |
| **Семантическое/текстовое** | «сообщение содержит X», «текст равен Y», «label — "Z"» | Сравнение текстового содержимого с эталоном |

**Обязательная процедура перед фиксацией PASS:**

1. Выпиши формулировку критерия дословно.
2. Определи тип свойства по таблице выше. Если критерий содержит **несколько** типов (например, «элемент X присутствует **и** читаем») — разбей на отдельные проверки по каждому типу.
3. Сравни с тем, что реально проверяет твоя верификация. Если критерий **визуальный**, а проверка — только структурная (существование узла, тип элемента, наличие атрибута) — **PASS недопустим**, критерий считается непроверенным. Нужна дополнительная проверка адекватного типа.
4. Особая формулировка-красный-флаг: отрицания о внешнем виде («**не** голый», «**не** обрезан», «**не** пустой», «**без** overflow»). Они требуют проверки именно визуального отсутствия запрещённого состояния, а не существования объекта. Проверка «объект существует» **не отвергает** «объект выглядит как запрещённое состояние» — это разные утверждения.

**Антипаттерн (тавтология критерия и проверки):** критерий требует визуального свойства объекта, а проверка подтверждает само существование того же объекта.

**Если адекватная проверка невозможна в доступном окружении** — зафиксируй критерий как невыполнимый с указанием инфраструктурного пробела в заметках, а не рапортуй PASS по ослабленной проверке.

### 5. Запись результата

К этому моменту секция Result уже содержит результаты по каждому пункту (записаны инкрементально на шаге 3). Осталось:

- Обновить/добавить **Что сделано** — краткое резюме всей работы
- Дополнить **Изменённые файлы** и **Заметки** если нужно
- **НЕ удалять и не переписывать** уже записанные результаты

> Шаблон Result: `templates/result-template.md`

Если секция Result ещё не создана — создай по шаблону:

```markdown
## Result

### Что сделано
- ...

### Изменённые файлы
- ...

### Заметки
- ...
```

### 6. ⛔ GATE — Финальная проверка перед завершением

**Перед выводом `---RESULT---` выполни ПОСЛЕДОВАТЕЛЬНО все проверки. При ЛЮБОМ нарушении — вернись к шагу 3 и исправь, не выводи RESULT.**

#### GATE-1: Проверка Edit-вызовов

За текущую сессию ты должен был вызвать инструмент **Edit** на файл тикета минимум дважды:
1. Для обновления чекбоксов DoD (`[ ]` → `[x]`)
2. Для записи секции Result

Если хотя бы один Edit не вызван — секция Result физически пуста. Это призрачное выполнение. Вернись к шагам 3–5.

#### GATE-2: Механическая Read-проверка файла тикета

Выполни `Read` файла тикета и убедись глазами:

1. **Ни одного `[ ]` в секции критериев готовности / DoD.** Все — `[x]` или с пометкой причины невыполнения.
2. **Секция `## Result` / `## Результат выполнения` физически заполнена** — не пустой шаблон.
3. **Frontmatter не модифицирован** — никаких новых ключей, никаких изменённых значений.

#### GATE-3: Self-check (мысленно, не в stdout)

Пройди по пунктам:
- Каждый пункт DoD отмечен `[x]` или зафиксирована причина невыполнения
- Если изменён код или тесты — проверочная команда запущена, вывод приложен к Result
- Секция Result содержит реальный контент, а не только заголовки шаблона
- Каждый выполненный пункт DoD имеет evidence в Result
- Все созданные/изменённые файлы перечислены в «Изменённые файлы»
- Все файлы из `context.files` доступны (без permission denied)
- Секция Result записана через **Edit** в файл тикета, а не только выведена в stdout
- Scope не расширен — изменены только файлы из DoD/context
- Поля `status` и `completed_at` не записаны в файл тикета
- Секция `## Ревью` не создавалась и не редактировалась
- Все временные файлы и пустые директории удалены

**При любой ошибке в GATE-1/2/3 — вернись к шагу 3 и выполни работу реально.**

## Выход

- Тикет обновлён секцией Result
- Все изменения внесены в файлы проекта
- Все три GATE пройдены успешно
- Тикет **НЕ** перемещён (перемещение — отдельный stage)

## Антипаттерны

- Создание тикетов/планов в `.workflow/` — запрещено
- **Создание тикетов при обнаружении дефекта** — зафиксируй в секции Result текущего тикета. НЕ создавай отдельный файл тикета.
- Перемещение тикета — запрещено
- Обновление `status`/`completed_at` в frontmatter — запрещено
- Работа без чтения контекстных файлов — приводит к неполному решению
- Завершение задачи без реального изменения файлов (если тикет требует изменения)
- **Призрачное выполнение** — вывод `---RESULT---` при пустой секции Result или неотмеченных DoD. Всегда проходи GATE 1-2-3 перед RESULT
- **Редактирование файла не по пути из `context.files`** — если тикет указывает конкретный путь к файлу, редактируй именно его, а не одноимённый файл в другой директории. Сверяй полный путь перед Edit
- **Вставка кода без понимания структуры** — перед Edit в существующий файл прочитай целевой участок (функцию, класс, блок) и пойми его границы. Вставляй код в правильное место согласно структуре, а не в произвольную точку
- **Пропуск проверки существующего прогресса** — перед началом работы всегда проверяй секции Result и Ревью тикета. Повторное выполнение без учёта сделанного — потеря времени и риск перезаписи
- **Отложенная запись результата** — записывай результат инкрементально после **каждого** пункта DoD. Запись всего в конце ведёт к потере прогресса при обрыве сессии и призрачному выполнению
