# Алгоритм: Проверка scope gap

Формализованный алгоритм определения, входит ли gap в scope декомпозиции.

## Вход

| Параметр | Тип | Описание |
|----------|-----|----------|
| `gap` | string | Описание одного конкретного пробела |
| `report` | file | Содержимое отчёта REPORT-{id}.md |
| `plan` | file | Содержимое исходного плана PLAN-{id}.md |

## Выход

| Параметр | Тип | Описание |
|----------|-----|----------|
| `in_scope` | boolean | true — создавать тикет, false — записать в «Новые требования» |
| `reason` | string | Обоснование решения |
| `check_failed` | string | Номер проверки, на которой gap был отклонён (если отклонён) |

## Алгоритм

```
FUNCTION check_scope(gap, report, plan) → {in_scope, reason, check_failed}

  # Шаг 1: Проверка источника
  source_section = найти секцию отчёта, откуда извлечён gap

  IF source_section IN [
    "Рекомендации для Architect",
    "План на следующий период",
    "Потенциальные риски",
    "Следующие шаги"
  ]:
    RETURN {false, "Gap из секции '{source_section}' — не является пробелом", "1"}

  IF source_section NOT IN [
    "Выполненные задачи",
    "Заблокированные задачи",
    "Риски и проблемы"
  ]:
    RETURN {false, "Gap из неизвестной секции '{source_section}'", "1"}

  # Шаг 2: Проверка принадлежности к плану
  matching_task = найти задачу в plan, которой соответствует gap

  IF matching_task == NULL:
    RETURN {false, "Нет соответствующей задачи в исходном плане", "2"}

  # Шаг 2a: Проверка exclusions из scope плана
  plan_exclusions = извлечь секцию «Исключено из scope» / «Excluded from scope» из plan

  IF plan_exclusions != NULL:
    FOR EACH exclusion IN plan_exclusions:
      IF gap описывает работу, подпадающую под exclusion:
        RETURN {false, "Gap подпадает под исключение из scope плана: '{exclusion}'", "2a"}

  IF gap описывает расширение scope (новый функционал, не заявленный в плане):
    RETURN {false, "Расширение scope за рамки плана", "2"}

  IF gap — deliverable другого тикета (создан как часть DoD):
    RETURN {false, "Deliverable другого тикета, не gap", "2"}

  # Шаг 3: Проверка статуса плана
  plan_status = извлечь status из plan

  IF plan_status == "completed":
    RETURN {false, "План завершён — gaps = новые инициативы", "3"}

  IF все ключевые задачи плана выполнены:
    RETURN {false, "Все ключевые задачи выполнены", "3"}

  # Шаг 4: Проверка workflow-управления
  IF gap содержит действия:
    - перемещение тикетов
    - изменение статуса других тикетов
    - запуск workflow-скриптов
    - операции с .workflow/tickets/, .workflow/plans/, .workflow/reports/
  :
    RETURN {false, "Содержит инструкции по workflow-управлению", "4"}

  # Все проверки пройдены
  RETURN {true, "Gap в scope: соответствует задаче '{matching_task}' плана", NULL}
```

## Пример применения

**Вход:**
- gap: "Не реализована валидация входных данных в модуле обработки"
- report: REPORT-003 (секция «Выполненные задачи», статус задачи: partial)
- plan: PLAN-006 (задача #3: "Реализовать валидацию входных данных", status: in_progress)

**Прохождение:**
1. Источник: «Выполненные задачи» → ✅ валидный источник
2. Принадлежность: задача #3 плана → ✅ есть в плане
3. Статус: plan status = in_progress → ✅ план не завершён
4. Workflow: нет управляющих инструкций → ✅

**Выход:** `{true, "Gap в scope: соответствует задаче #3 плана", NULL}`

## Граничные случаи

| Случай | Решение |
|--------|---------|
| Gap на стыке двух задач плана | Создать один тикет, ссылаясь на обе задачи |
| Gap частично в scope, частично нет | Разделить: in-scope часть → тикет, out-of-scope → «Новые требования» |
| Заблокированная задача по внешней зависимости | Не gap — внешние зависимости не контролируются скилом |
| Gap описывает «улучшить» вместо «исправить» | Проверить, было ли в DoD плана — если нет, out of scope |
