# Валидация scope для gaps

Критерии определения, какие gaps входят в scope декомпозиции, а какие — нет.

## Проверка 1: Источник gap

Gap — это исключительно то, что **не выполнено или выполнено некорректно** из явного списка задач плана.

### Валидные источники gaps

| Секция отчёта | Что является gap |
|---------------|-----------------|
| «Выполненные задачи» | Задачи со статусом failed/skipped, где DoD не выполнен |
| «Заблокированные задачи» | Задачи, заблокированные по техническим причинам (не по внешним зависимостям) |
| «Риски и проблемы» | Только конкретные сбои в выполненных задачах |

### НЕ являются источником gaps

| Секция отчёта | Почему не gap |
|---------------|--------------|
| «Рекомендации для Architect» | Советы, не пробелы |
| «План на следующий период» | Новые инициативы, не пробелы |
| «Потенциальные риски» | Гипотезы, не пробелы |
| Упоминания будущих задач (XXX-005 и т.п.) | Планируемая работа, не текущие пробелы |

## Проверка 2: Принадлежность к scope исходного плана

### Gap в scope (создавать тикет)

- Задача была в исходном плане, но не выполнена или выполнена некорректно
- Доработка существующего результата, который не соответствует DoD плана
- Исправление ошибки в уже сделанной работе по плану

### Gap вне scope (НЕ создавать тикет)

- Новая работа, которой не было в исходном плане
- Расширение scope за рамки первоначальных требований
- Задачи, созданные другими тикетами как часть их DoD (deliverables, не gaps)
- Задачи из секций «Рекомендации», «План на следующий период», «Следующие шаги»

## Проверка 3: Статус плана

| Статус плана | Действие |
|-------------|----------|
| `in_progress` | Декомпозировать gaps нормально |
| `completed` | **НЕ создавать тикеты** — gaps в завершённом плане = новые инициативы для следующего плана |
| Все ключевые задачи выполнены | **НЕ создавать тикеты** — аналогично completed |

## Проверка 4: Запрет на workflow-управление

**⛔ КРИТИЧЕСКИ ВАЖНО:** Тикеты **НИКОГДА** не должны содержать инструкции по управлению workflow-артефактами.

### Запрещено создавать тикеты на:

- Перемещение других тикетов между папками (backlog → ready → done)
- Изменение статуса или метаданных других тикетов
- Запуск workflow-скриптов (`move-ticket.js`, `pick-next-task.js` и т.п.)
- Любые операции с `.workflow/tickets/`, `.workflow/plans/`, `.workflow/reports/`

### Если обнаружен «застрявший» тикет:

Это инфраструктурная проблема → записать в «Новые требования (вне scope)» для ручного разбора.
