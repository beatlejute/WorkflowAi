## План действий для QA-117

### До запуска

1. **Прочитать файл теста** `tests/manual/login-flow.spec.ts` — найти строку с hardcoded путём `reports/QA-009-...` и убедиться, что понимаю все места, где фигурирует старый prefix.

2. **Обновить prefix в файле теста** — заменить `QA-009` на `QA-117` через Edit **до запуска**. Это критично: ручное переименование артефактов *после* запуска — антипаттерн, который создаёт гонку `mtime` между файлами и `updated_at` тикета и ломает artifact-snapshot diff верификатора. Правильный порядок: сначала правка файла, потом запуск.

3. **Убедиться, что директория `reports/` существует** — если нет, создать.

### Во время запуска

4. **Запустить тест** — выполнить `tests/manual/login-flow.spec.ts` штатным способом (через Playwright или иным раннером, который использует проект).

### После запуска

5. **Убедиться, что артефакт сохранён с правильным именем** — проверить наличие файла `reports/QA-117-screenshot-*.png` (а не `QA-009-*`).

6. **Открыть скриншот через Read и провести self-review** (принцип 8: Visual TC = Screenshot + Self-Review) — описать одним предложением, что видно на скриншоте, до записи PASS.

7. **Зафиксировать результат в тикете** — статус PASS/FAIL, a11y assertion или описание наблюдения, ссылка на `reports/QA-117-screenshot-*.png` + описание содержимого скриншота.

8. **Cleanup** — удалить любые промежуточные/отладочные файлы из `reports/`, если появились; в `reports/` должны остаться только файлы, на которые ссылается тикет QA-117.

---

**Ключевое правило:** обновление prefix происходит **в файле теста перед запуском**, не после — это единственный способ гарантировать корректный `mtime` артефактов и пройти верификацию.
