## План действий для QA-117

### До запуска теста

Согласно правилу скила («Переиспользование существующего теста»):

> Если тест уже существует и сохраняет артефакты с hardcoded prefix другого тикета — **обнови prefix на ID текущего тикета в тест-файле перед запуском**.

1. Прочитать файл `tests/manual/login-flow.spec.ts`
2. Найти строку с hardcoded `QA-009` (например: `screenshotPath = 'reports/QA-009-...'`)
3. Заменить `QA-009` на `QA-117` прямо в файле теста с помощью Edit
4. Убедиться, что директория `reports/` существует

Ручное переименование файлов **после** запуска — антипаттерн: он создаёт гонку `mtime` между артефактами и `updated_at` тикета, что ломает artifact-snapshot diff верификатора.

---

### Во время запуска

5. Запустить тест (например, `npx ts-node tests/manual/login-flow.spec.ts` или через соответствующий runner)
6. Следить за выводом — зафиксировать статус (PASS / FAIL) и путь сохранённого скриншота

---

### После запуска

7. Проверить, что артефакт сохранён под именем `reports/QA-117-screenshot-*.png` (а не `QA-009-*`)
8. Открыть скриншот через Read и визуально убедиться, что он содержит нужный контент
9. Зафиксировать результат в тикете: статус TC, a11y/evidence, ссылка на PNG + краткое описание содержимого
10. Заполнить `completed_at` в frontmatter тикета и итоговый отчёт по шаблону `templates/test-session-report.md`

---

**Ключевое правило:** prefix меняется в исходнике теста **до** запуска, а не через переименование файла после — это единственный корректный способ, гарантирующий консистентность артефактов.
