## Стратегия тестирования VSCode-расширения

### Инструмент: Windows-MCP

VSCode-расширение — это desktop-приложение, поэтому согласно таблице выбора инструмента:

| Критерий | Выбор |
|----------|-------|
| Тип приложения | Desktop (VSCode-расширение) |
| Основной инструмент | **Windows-MCP** |
| Knowledge-модули | `knowledge/desktop-tools-core.md` + `knowledge/sandbox-core.md` |

### Среда запуска: Windows Sandbox (обязательно)

**⛔ Прямое тестирование на хосте недопустимо** — по двум причинам:
1. Загрязняет рабочую среду разработчика (установленные расширения, настройки VSCode, глобальные конфиги)
2. Не воспроизводит чистое состояние — баги, связанные с первым запуском или чистой установкой, будут пропущены

Вся работа ведётся **исключительно через Sandbox**.

### Порядок подготовки

```
1. Прочитать ../shared/README.md (если есть) — проектные пути и конфиги
2. Загрузить knowledge/sandbox-core.md — quick-start Sandbox, персистенция evidence
3. Загрузить knowledge/desktop-tools-core.md — команды Windows-MCP, паттерны навигации
4. При необходимости: knowledge/desktop-tools-advanced.md — Snapshot (a11y tree), DevTools
```

### Схема работы

```
Sandbox (изолированная среда)
  └── VSCode с установленным .vsix
        └── Windows-MCP → управление UI, Snapshot (a11y tree)
              └── evidence → reports/ (через персистенцию из Sandbox)
```

### Ключевые ограничения

- **Evidence эфемерны внутри Sandbox** — файлы не переживают закрытие сессии; нужно явно выносить скриншоты/логи на хост через механизм, описанный в `sandbox-core.md`
- **Бюджет Snapshot** — каждый вызов a11y tree стоит дорого; при < 30% бюджета переключиться на text-assertion вместо скриншотов (кроме визуальных TC)
- **MCP-disconnect** — при обрыве соединения загрузить `knowledge/sandbox-advanced.md` → перезапуск сессии
