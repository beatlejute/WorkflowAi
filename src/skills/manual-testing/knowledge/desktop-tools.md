# Инструменты работы с desktop-приложениями

Справочник по инструментам и командам для тестирования desktop-приложений (VSCode, Electron-приложения и др.) через Windows-MCP.

## Windows-MCP — основной инструмент

Windows-MCP — MCP-сервер для управления Windows GUI через Accessibility Tree (UI Automation). Агент взаимодействует с элементами интерфейса напрямую через a11y-идентификаторы, без computer vision.

**Преимущества:**
- Быстрое и надёжное взаимодействие (0.2–0.9 сек на действие)
- Работает с любым Windows-приложением (VSCode, Electron, WPF, WinForms)
- Не требует vision-модели — использует Accessibility Tree
- Поддерживает мультимониторные конфигурации

### Установка

**Требования:** Python 3.13+, UV package manager

```bash
pip install uv
```

**Конфигурация MCP-клиента (Claude Desktop):**

```json
{
  "mcpServers": {
    "windows-mcp": {
      "command": "uvx",
      "args": ["windows-mcp"]
    }
  }
}
```

**Конфигурация для Claude Code (Windows):**

```json
{
  "mcpServers": {
    "windows-mcp": {
      "command": "cmd",
      "args": ["/c", "uvx", "windows-mcp"]
    }
  }
}
```

> **Важно:** На Windows для Claude Code необходим враппер `cmd /c` перед `uvx`, иначе будет ошибка "Connection closed".

### Захват состояния экрана

| Команда | Описание | Параметры | Когда использовать |
|---------|----------|-----------|-------------------|
| `screenshot` | Быстрый скриншот рабочего стола с позицией курсора и списком активных окон | `display: [0]` (опционально, индекс монитора) | По умолчанию первый вызов — когда нужен визуальный контекст |
| `snapshot` | Полный захват состояния: accessibility tree с ID интерактивных элементов, скроллируемые области, скриншот | `use_vision: bool`, `use_dom: bool`, `display: [0]` | Когда нужны ID элементов для кликов, анализ структуры UI |

**Разница:** `screenshot` — быстрый (только картинка), `snapshot` — полный (a11y tree + картинка). Начинай с `screenshot`, переключайся на `snapshot` когда нужны координаты элементов.

### Взаимодействие с элементами

| Команда | Описание | Параметры |
|---------|----------|-----------|
| `click` | Клик по координатам или label | `loc: [x, y]` (координаты) ИЛИ `label: int` (ID из Snapshot). Опционально: `button: "left"/"right"`, `double: true` |
| `type` | Ввод текста в элемент | **Обязательно:** `loc: [x, y]` ИЛИ `label: int`. `text: string`, `press_enter: "true"/"false"` (опционально), `clear: true` (опционально — очистить перед вводом) |
| `scroll` | Прокрутка вертикально/горизонтально | `direction: "up"/"down"/"left"/"right"`, `wheel_times: int` (количество прокруток). **Используй `loc: [x, y]`** (координаты центра скроллируемой области). `label: int` ненадёжен — ID устаревают, используй loc. Опционально: `type: "horizontal"` |
| `move` | Перемещение мыши / перетаскивание | `loc: [x, y]`, `drag: bool` (опционально) |
| `shortcut` | Нажатие клавиатурных комбинаций | `shortcut: string` (например `"ctrl+shift+p"`, `"Escape"`) |
| `wait` | Пауза | `duration: float` (секунды) |

> **⚠️ ВАЖНО:** `type`, `click` и `scroll` требуют `loc` или `label`. Вызов без них вернёт ошибку. Получи координаты/ID через `snapshot` перед вызовом. **Для `scroll` предпочитай `loc`** — координаты центра скроллируемой области стабильны, а label-ID устаревают между Snapshot-ами и дают ошибку «Label N out of range».

> **⚠️ Типы параметров:** `loc` — это JSON-массив `[x, y]`, НЕ строка `"[x, y]"`. `label` — int, НЕ строка. `wheel_times` — int. Передавай их как нативные типы, не оборачивай в кавычки.

## Паттерны тестирования desktop-приложений

### Запуск VSCode с расширением

```
1. shell → "code --extensionDevelopmentPath=d:/Dev/workflowAiVsCode"
2. screenshot → начальное состояние
3. Проверить: VSCode открылся? Расширение активировано?
```

### Открытие Command Palette и выполнение команды

```
1. snapshot → получить состояние окна VSCode, найти label N для input-поля Command Palette
2. shortcut → "Ctrl+Shift+p" (открыть Command Palette)
3. type → text: "workflowAI: Show Pipeline", label: N (или loc из Snapshot)
4. shortcut → "Enter" (выполнить)
5. screenshot → зафиксировать результат
```

### Проверка TreeView расширения

```
1. snapshot → получить a11y tree (найти TreeView в sidebar)
2. click → label N иконки расширения в Activity Bar
3. snapshot → получить элементы TreeView
4. click → label N элемента в TreeView
5. screenshot → зафиксировать состояние
6. Проверить: элементы отображаются? Данные корректны?
```

### Проверка StatusBar

```
1. snapshot → получить a11y tree
2. Найти StatusBar-элемент расширения по тексту
3. click → label N StatusBar-элемента
4. screenshot → зафиксировать реакцию
5. Проверить: подсказка корректна? Клик вызывает нужное действие?
```

### Проверка Output Channel

```
1. shortcut → "Ctrl+Shift+u" (открыть Output panel)
2. snapshot → получить содержимое панели, найти label N dropdown-а
3. click → label N dropdown выбора канала
4. type → text: "WorkflowAI", label: N (label input-поля из Snapshot)
5. screenshot → зафиксировать логи
6. Проверить: логи присутствуют? Нет ошибок?
```

### Проверка уведомлений (notifications)

```
1. Выполнить действие, вызывающее уведомление
2. screenshot → зафиксировать уведомление
3. Проверить: уведомление появилось? Текст корректен?
4. shortcut → "Escape" (закрыть уведомление)
```

### Проверка WebView панели расширения

```
1. Открыть WebView через Command Palette или TreeView
2. snapshot → use_dom: true (извлечь DOM WebView)
3. screenshot → визуальное состояние
4. Проверить: контент загружен? Стили корректны?
```

## Работа с Accessibility Tree

Windows-MCP использует Windows UI Automation для доступа к элементам интерфейса. `snapshot` возвращает дерево с:
- **ID элементов** — для точных кликов
- **Имена и роли** — для идентификации элементов (Button, TreeItem, MenuItem)
- **Координаты** — для click/type
- **Состояния** — enabled/disabled, expanded/collapsed, checked/unchecked

### Типичные элементы VSCode в a11y tree

| Элемент VSCode | Роль в a11y tree | Как найти |
|---------------|-----------------|-----------|
| Command Palette | ComboBox / List | После `Ctrl+Shift+p` |
| Activity Bar | ToolBar | Левая панель с иконками |
| Sidebar (Explorer/Extensions) | Tree / TreeItem | После клика на Activity Bar |
| Editor tabs | TabItem | Верхняя область редактора |
| StatusBar items | Button / Text | Нижняя полоса окна |
| Notifications | Group / Button | Правый нижний угол |
| Output panel | Text / Document | Нижняя панель |

## Эффективность MCP-вызовов

### Label-based навигация (ПРИОРИТЕТ)

**Всегда предпочитай `Click(label=N)` вместо `Click(loc=[x,y])`.**

Координаты нестабильны — collapse/expand секции, переключение табов, resize окна, открытие/закрытие панелей сдвигают все элементы. Label-ID из Snapshot (`# id` — первый столбец в таблице Interactive Elements) автоматически привязан к элементу.

**⚠️ ВАЖНО: параметр `label` принимает ТОЛЬКО числовой ID (int).** Текстовые имена (`Click(label="Run Pipeline")`) вызовут ошибку `int_parsing`. Число берётся из столбца `# id` в выводе Snapshot.

```
# Snapshot выдаёт:
# 61|...|button|Run Pipeline (Ctrl+Shift+W R)|(376,101)|...
# 62|...|button|Stop Pipeline|(376,101)|...
# 63|...|tree_item|TICKETS Section|(244,305)|...

# ✅ ХОРОШО — числовой ID из Snapshot
Click(label=61)    # Run Pipeline
Click(label=63)    # TICKETS Section

# ❌ ПЛОХО — текстовая строка, вызовет ошибку int_parsing
Click(label="Run Pipeline (Ctrl+Shift+W R)")
Click(label="TICKETS Section")

# ❌ ПЛОХО — координаты, сломаются при resize/collapse
Click(loc=[244, 305])
```

**Когда label недоступен** (элемент без `# id` в a11y tree) — используй координаты, но **только из последнего Snapshot**. Не переиспользуй координаты после действий, меняющих layout (collapse/expand, открытие панели, resize).

**⛔ Label-ID ОДНОРАЗОВЫЕ.** Числовой `label: N` валиден ТОЛЬКО для немедленного использования после Snapshot, в котором он получен. Любое действие между Snapshot и Click/Type/Scroll (другой Click, Screenshot, PowerShell, даже Wait) может сдвинуть ID. **При ошибке «Label N out of range» — не повторяй с тем же N, сделай новый Snapshot и найди элемент заново по текстовому имени.**

**Паттерн навигации — запоминай текстовое имя, ищи числовой ID:**
```
# Запомни: "TICKETS Section → More Actions..."
# При каждом Snapshot ищи ID этих элементов по имени в таблице
# и передавай числовой ID в Click(label=N)
```

### Snapshot budget

Snapshot — самый дорогой вызов (возвращает ~300-500 строк a11y tree, ~5-10k tokens). Используй его экономно:

| Когда делать Snapshot | Когда НЕ делать |
|----------------------|-----------------|
| Первый раз — для ориентации в UI | После Escape / закрытия меню (layout не изменился) |
| После действия, меняющего **набор элементов** (новое меню, диалог) | После Wait (если не ожидаешь изменения UI) |
| Для верификации появления/исчезновения элемента | Если нужный label уже известен из предыдущего Snapshot |
| После смены вкладки / открытия панели | Для «проверки что ничего не сломалось» |

**Правило:** после первого Snapshot запомни **текстовые имена** элементов и их **числовые ID** (не координаты). Далее используй `Click(label=N)` с числовым ID. Если между Snapshot и Click прошло несколько действий — сделай свежий Snapshot и найди элемент по имени заново. Новый Snapshot нужен только когда появляются **новые элементы**, которых не было раньше (меню, диалог, новая панель), или когда прошлый ID мог устареть.

**⚠️ Context overflow:** каждый Snapshot добавляет ~5-10k tokens в контекст. При бюджете ~100-140 MCP-вызовов чрезмерное использование Snapshot вызывает context overflow и вынужденный restart сессии с потерей контекста. Целевой лимит: **максимум 1-2 Snapshot на тест-кейс**.

**Параметр `use_vision`:** по умолчанию используй `use_vision: false` — это экономит токены (vision-обработка скриншота не выполняется, возвращается только a11y tree). Используй `use_vision: true` только когда нужна **визуальная интерпретация** скриншота (верификация evidence, проверка визуальных артефактов, чтение текста, недоступного через a11y tree).

### Минимальный текстовый вывод (Minimal Thinking)

Каждое текстовое сообщение между MCP-вызовами расходует токены контекстного окна. При бюджете 70-100 MCP-вызовов многословные рассуждения ускоряют context overflow и приводят к обрыву сессии.

```
# ❌ ПЛОХО — 3 строки рассуждений перед каждым действием
"Pipeline запущен, 8 секунд прошло. Делаю скриншот для верификации Running-состояния.
Stage-beta должен быть активен, проверю StatusBar на наличие sync~spin иконки:"
Screenshot(...)

# ✅ ХОРОШО — 1 строка, action-oriented
"TC-005: проверяю Running state"
Screenshot(...)
```

**Правило:** между MCP-вызовами — максимум 1 короткая строка (что делаю и зачем). Анализ, выводы, описание багов — в тикет при фиксации результатов.

### Batch-действия

Группируй 2-3 MCP-вызова перед проверочным Snapshot. Каждый цикл «действие → Wait → Snapshot» стоит ~30-40 секунд (MCP latency + LLM обработка).

**Плохо:**
```
Click(tab) → Wait(1s) → Snapshot → Click(button) → Wait(1s) → Snapshot
```

**Хорошо:**
```
Click(tab) → Click(button) → Snapshot (проверка результата)
```

Используй промежуточные Snapshot только если следующее действие зависит от координат, которых ещё нет.

### Прокрутка списков и dropdown

Snapshot показывает `vertical_scroll_percent` для скроллируемых элементов. Если scroll_percent > 0 — над видимой областью есть элементы. Если scroll_percent < 100 — под видимой областью есть элементы. Перед выводом «элемент отсутствует» — прокрути **в обоих направлениях**:

```
1. Snapshot → найти scrollable element, проверить vertical_scroll_percent
2. Если > 0 → Scroll(direction: "up", loc: [x, y]) до начала (scroll_percent = 0)
3. Snapshot → проверить, появился ли искомый элемент
4. Если не найден и scroll_percent < 100 → Scroll(direction: "down") до конца
5. Snapshot → проверить снова
6. Только после полной прокрутки в обе стороны можно утверждать «элемента нет»
```

**Антипаттерн:** увидеть N элементов dropdown и решить, что остальных нет. Dropdown может быть открыт в **середине** или **конце** списка — элемент может быть **выше** текущей видимой области. Пример: Output channel dropdown в VSCode сортирует каналы по алфавиту — канал на букву «A» будет выше канала на букву «W», и может быть за пределами видимой области.

### Проверка Developer Console (DevTools)

При проверке DevTools Console обращай внимание на **счётчик ошибок** в заголовке (`🔴 N errors, ⚠️ M warnings`), а не только на видимые строки в логе. Видимая область может показывать лишь последние записи, а ошибки находятся выше.

```
1. Shortcut → "Ctrl+Shift+I" (открыть DevTools)
2. Snapshot → прочитать счётчик ошибок в заголовке Console
3. Если счётчик > 0 — прокрутить лог вверх и проверить каждую ошибку
4. Отличить ошибки расширения от системных (Chromium, телеметрия VSCode)
5. PASS только если ошибок расширения = 0 (системные ошибки не считаются)
```

**Антипаттерн:** увидеть одну красную строку внизу лога и проигнорировать счётчик «19 errors» в заголовке.

### Window management

Если `Shortcut` не закрыл/переключил окно:

1. **Не повторяй тот же Shortcut** — если не сработал первый раз, повтор не поможет
2. **App(switch)** — переключись на нужное окно: `App(action: "switch", app_name: "Visual Studio Code")`
3. **Click по title bar** — кликни по заголовку целевого окна для перефокусировки
4. **Click по X** — для закрытия: найди кнопку «Закрыть» через Snapshot и кликни по координатам

## Советы

- **Артефакты — в `reports/`** — все скриншоты и файлы сохраняй в `reports/` в корне проекта, не в корень
- **Snapshot-first в Sandbox** — в изолированных средах (Windows Sandbox) начинай с `snapshot` (a11y tree + скриншот), а не `screenshot`. В Sandbox визуальная интерпретация скриншотов ненадёжна: низкое разрешение, задержки отрисовки. A11y tree даёт точные label-имена элементов без угадывания. **После первого Snapshot переходи на label-based Click** — не делай повторный Snapshot для каждого действия
- **На хосте — screenshot → snapshot** — на рабочей машине можно начинать с `screenshot` для быстрого контекста, переходя на `snapshot` когда нужны label-имена элементов
- **Верификация перед действием** — перед кликом, вводом текста или прокруткой **убедись через Snapshot (a11y tree), что целевой элемент реально существует**. Не полагайся на визуальную интерпретацию screenshot — это приводит к кликам «в пустоту» и галлюцинациям (например, попытки закрыть несуществующее контекстное меню)
- **Не используй `Wait` в Sandbox** — между MCP-вызовами проходит 3-10 секунд (MCP latency + LLM processing). Этого достаточно для обновления UI. `Wait` — пустая трата MCP-бюджета. Исключение: ожидание длительного процесса (запуск pipeline ~30s), где нужна конкретная задержка
- **`shortcut` для навигации** — Command Palette (`Ctrl+Shift+p`) быстрее и надёжнее, чем поиск кнопок мышью
- **`shell` / `PowerShell` для подготовки** — создание тестовых файлов, запуск VSCode с параметрами
- **Проверяй Output Channel** — расширения логируют ошибки туда, а не в console
- **Мультимонитор** — используй `display: [0]` или `display: [1]` для выбора монитора
- **`App(mode: "switch")` для перефокусировки** — если окно потеряло фокус или UI не реагирует на Escape/Click, переключись на целевое окно через `App(mode: "switch", name: "...")`

## Выбор MCP-сервера: хост vs Sandbox

Инструменты одинаковые (`screenshot`, `click`, `type` и т.д.), но вызываются через разные MCP-серверы в зависимости от среды:

| Среда | MCP-сервер | Вызов инструмента |
|-------|-----------|-------------------|
| **Хост** (рабочая машина) | `windows-mcp` | `mcp__windows-mcp__screenshot` |
| **Windows Sandbox** (изолированная среда) | `sandbox-desktop` | `mcp__sandbox-desktop__screenshot` |

**Как определить:** проверь `.mcp.json` проекта — если настроен `sandbox-desktop` с URL на Sandbox, используй его для изолированного тестирования. Подробности — в `knowledge/sandbox-testing.md`.

> **Важно:** в Sandbox **не используй `Wait`** между обычными UI-действиями. Латентность MCP (3-10 сек между вызовами) уже превышает время обновления UI. Каждый `Wait` — потеря MCP-вызова из бюджета.

## Ограничения

- Выделение текста внутри параграфа ограничено (a11y tree не всегда гранулярен до символа)
- Windows-язык должен быть English для корректной работы `app` (launch)
- Латентность MCP: 3-10 сек между вызовами (network + LLM processing) — достаточно для обновления UI, `Wait` не нужен
- **Модификаторы + Click невозможны:** MCP Click не поддерживает Ctrl+Click, Shift+Click, Alt+Click. Это значит: Document Links (Ctrl+Click по ссылке в VSCode), мульти-выделение (Ctrl+Click) и расширенное выделение (Shift+Click) **недоступны через MCP**

### Альтернативы для Ctrl+Click (Document Links, Go to Definition)

| Задача | Альтернатива |
|--------|-------------|
| Открыть ссылку в редакторе | Command Palette → `Go to Definition` (F12) |
| Перейти по Document Link | Command Palette → `Open Link` |
| Открыть файл по пути | `Shortcut(shortcut: "ctrl+p")` → `Type(text: "путь")` → Enter |

**Если TC требует Ctrl+Click — отметь BLOCKED** с причиной «MCP не поддерживает модификаторы + Click» и предложи альтернативный подход.
