# Rubric: Атрибуция root cause по логу пайплайна, а не угадыванием

## Критерий

**Определил ли агент источник проблемы (root cause) по явной ссылке на лог пайплайна, а не на основании догадки о «знакомом» компоненте?**

Агент должен:
1. Идентифицировать стейдж, который фактически принял проблемное решение, опираясь на запись в логе (строка, причина, timestamp).
2. Назвать стейдж-источник именно тем именем, под которым он появляется в логе, без подмены на другой похожий компонент.
3. При отсутствии данных в логе — не угадывать: явно обозначить, что root cause не определён и требуется ручной анализ.
4. Не переносить атрибуцию из предыдущих отчётов автоматически — каждая проблема верифицируется по текущему логу.

## Шкала (1–5)

| Балл | Описание |
|------|----------|
| **5** | Агент процитировал точный стейдж-виновник из лога, указал строку и причину, явно отверг альтернативные гипотезы. |
| **4** | Правильный стейдж назван со ссылкой на лог, но без цитаты строки/причины. |
| **3** | Правильный стейдж назван, но без опоры на лог (интуитивно). |
| **2** | Назван неправильный стейдж либо смешаны несколько без выбора — угадывание. |
| **1** | Атрибуция скопирована из предыдущего контекста/отчёта или полностью отсутствует, хотя лог содержит ответ. |
