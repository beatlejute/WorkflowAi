# Воркфлоу: STANDARD — Стандартный отчёт об итерации

Формирование отчёта о выполненных задачах за итерацию.

## Алгоритм выполнения

### 1. Собрать информацию

Прочитать все тикеты из:
- `.workflow/tickets/done/` — выполненные
- `.workflow/tickets/blocked/` — заблокированные
- `.workflow/tickets/in-progress/` — в работе

### 2. Проверка аномалий: in-progress тикеты с заполненным результатом

**Автоматическая проверка (рекомендуется):**

```bash
node .workflow/src/scripts/check-anomalies.js
```

Скрипт выводит результат в формате:
```
---RESULT---
status: ok|anomalies_found|error
anomalies_count: N
anomalies: [{"id": "IMPL-001", "title": "...", "recommendation": "..."}]
---RESULT---
```

**Ручная проверка (если скрипт недоступен):**

Для каждого тикета в `.workflow/tickets/in-progress/`:
1. Прочитать содержимое файла
2. Найти раздел `## Результат выполнения` (или `## Result`)
3. Проверить подраздел `### Summary` (или `### Что сделано`)
4. Если подраздел содержит непустой текст (не только `<!-- ... -->` комментарии) — это аномалия

**Действия при обнаружении аномалии:**
- Добавить тикет в секцию "Аномалии" отчёта
- Указать рекомендацию: "Проверьте тикет и переместите в done/ или review/ если выполнен"

### 3. Определить период

```
Период: от последнего отчёта до сейчас
Если отчётов нет — от создания первого тикета
```

### 4. Для каждой выполненной задачи

Извлечь:
- ID и название
- Время выполнения (created_at → completed_at)
- Результат (из секции Result)
- Проблемы (если были)

### 5. Рассчитать метрики

**Автоматический расчёт (рекомендуется):**

Если известен ID активного плана (`PLAN-NNN`):

```bash
node .workflow/src/skills/create-report/scripts/calc-metrics.js <PLAN-NNN>
```

Скрипт выводит результат в формате:
```
---RESULT---
{
  "plan_id": "PLAN-NNN",
  "total_tickets": N,
  "velocity": { "done_count": N, "days_elapsed": N, "velocity_day": N, "velocity_week": N },
  "plan_health": { "total_tickets": N, "done_tickets": N, "completion_pct": N, "expected_pct": N, "delta": N, "health_status": "ON_TRACK|AT_RISK|OFF_TRACK" },
  "distribution": { "<type>": { "count": N, "pct": N } },
  "anomalies": [{ "type": "...", "severity": "...", "message": "..." }]
}
---RESULT---
```

Используй готовые данные из JSON для заполнения отчёта. Не пересчитывай метрики вручную.

**Ручной расчёт (если скрипт недоступен):**

Загрузить `knowledge/report-metrics.md` для формул и метрик.

Рассчитать:
- Количество задач по статусам (done, in-progress, blocked, ready, backlog)
- Распределение по типам
- Velocity (задач / единица времени)
- Процент выполнения плана

### 6. Выделить проблемы и аномалии

**Проблемы:**
- Для каждой заблокированной задачи: причина блокировки, как разблокировать
- Для задач с problems: описание проблемы, как была решена
- Для каждого пропущенного (`⏭ skipped`) тикета: **точная атрибуция root cause** (см. шаг 6.1)

**Аномалии:**
- In-progress тикеты с заполненным результатом (из шага 2)

### 6.1. Атрибуция root cause проблем по логу пайплайна

> **⛔ ОБЯЗАТЕЛЬНЫЙ ШАГ.** Без него отчёт содержит угаданные обвинения, а не факты. Это нарушает evidence-based анализ и приводит к правке не тех компонентов.

**Зачем:** тикет фиксирует только своё состояние (`⏭ skipped`, `❌ failed`, `blocked`), но **не указывает, какой именно стейдж пайплайна принял это решение**. Без чтения лога ты вынужден угадывать виновника. Угадывание = ошибка атрибуции = правка не того компонента.

**Алгоритм:**

1. **Найди лог активной сессии.** В директории `.workflow/logs/` лежат файлы вида `pipeline_YYYY-MM-DD_HH-MM-SS.log`. Возьми **самый свежий** лог по дате (mtime), относящийся к анализируемому периоду.

2. **Для каждой проблемы из шага 6** (skipped, failed, blocked, аномалии):
   1. Найди в логе строки с упоминанием `ticket_id` проблемного тикета (Grep по `{TICKET-ID}`).
   2. Найди стейдж, который **последним** изменил статус тикета на проблемный. Ищи паттерны:
      - `[stage-name] COMPLETE stage="..." status="irrelevant|failed|blocked|..."` — кто принял решение
      - `[stage-name] OUTPUT ↓ ... reason: ...` — обоснование решения
      - `[move-ticket] ... ticket_id: X ... to: done|blocked|...` — последнее перемещение
   3. Извлеки **точное имя стейджа** из лога и его **точную причину** (как написано в OUTPUT).

3. **Запиши в отчёт точную атрибуцию:**
   - Имя стейджа из лога (например, `check-relevance` — это **скил**, исполняемый агентом; `check-conditions` — это **скрипт** `.workflow/src/scripts/check-conditions.js`). Не путай эти два класса компонентов.
   - Номер строки лога, где принято решение.
   - Точную причину как процитировано в логе.

4. **Формат записи в отчёт:**
   ```
   - Тикет XXX-NNN: ⏭ skipped стейджем `check-relevance` в строке логa NNNN
     с причиной `dependencies_inactive`. Лог: pipeline_YYYY-MM-DD_HH-MM-SS.log:NNNN
   ```

> **⛔ Антипаттерн «копирование root cause из предыдущих отчётов».** Если в REPORT-(N-1) проблема была атрибутирована стейджу X, **не копируй** эту атрибуцию в новый отчёт автоматически. Каждый инцидент проверяй заново по своему логу: возможно, в REPORT-(N-1) была ошибка атрибуции, и копирование её увековечит. Каждый отчёт = независимая верификация по логу.

> **⛔ Антипаттерн «угадывание по знакомым именам».** Если ты не нашёл в логе строку с решением — **не пиши** «возможно, это X». Напиши: `root cause не определён, требуется ручной анализ лога`. Это честнее и не уводит коуча/стейкхолдера в неправильную сторону.

**Граничные случаи:**

| Случай | Действие |
|--------|----------|
| Лог сессии не найден в `.workflow/logs/` | В отчёте: «Лог сессии недоступен, root cause проблем не атрибутирован». Не угадывай. |
| Тикет упоминается в логе, но решение принял неизвестный стейдж | Цитируй имя стейджа как есть, даже если оно тебе незнакомо. Не интерпретируй. |
| В логе несколько `OUTPUT` для одного тикета — какой выбрать? | Бери **последний по времени** перед сменой статуса (см. timestamp в начале строки). |
| Лог огромный (>250КБ) — Read падает | Используй Grep с фильтром по `ticket_id`, не читай файл целиком. |

### 7. Связать с планом

Если есть активный план:
- Какой процент выполнен
- Какие задачи из плана закрыты
- Что осталось

### 8. Сохранить отчёт

1. Прочитай шаблон: `.workflow/templates/report-template.md`
2. Определи следующий ID — вызови скрипт генерации:
   ```bash
   node .workflow/src/scripts/get-next-id.js --prefix REPORT --dir .workflow/reports
   ```
   Прочитай JSON из `---RESULT---`, возьми поле `id` (формат `REPORT-NNN`).

   > **Почему скрипт обязателен:** скрипт рекурсивно сканирует **все** поддиректории `.workflow/reports/` (включая возможные `archive/` или подпапки по периодам). Ручной поиск через Glob может пропустить часть файлов → возникает collision с существующими ID, ломаются ссылки между отчётами.

   > **⛔ Не используй ручной алгоритм как fallback.** Если скрипт недоступен — это инфраструктурная ошибка, а не повод изобретать ID самостоятельно. Сообщи о проблеме и **остановись**.

3. Сохрани в: `.workflow/reports/REPORT-{NNN}.md`

### 9. Вывести результат

```
---RESULT---
status: default
report_id: REPORT-{NNN}
---RESULT---
```
