## Шаг 9.B: СТОП-гейт аллокации ID

Проверяю входной параметр `id_ranges_json`:

**Параметр `id_ranges_json` на входе стадии отсутствует.** Согласно воркфлоу (шаг 9.B), это инфраструктурная ошибка — стадия `allocate-ticket-ids` должна была выделить диапазоны ID до запуска декомпозиции и передать их в виде JSON-строки.

**⛔ СТОП.** Не продолжаю декомпозицию.

**Причина:** источник нумерации тикетов — **только** `id_ranges_json`, полученный от инфраструктуры (стадия `allocate-ticket-ids`, которая внутри использует `get-next-id.js` для рекурсивного сканирования всех поддиректорий `.workflow/tickets/`). Агент-декомпозитор:
- Не вызывает `get-next-id.js` самостоятельно
- Не сканирует `.workflow/tickets/` для определения следующего ID
- Не вычисляет «максимальный существующий номер + 1»
- Не начинает нумерацию с 001

ID должны быть выделены машиной и переданы через `id_ranges_json` вида `{"IMPL":6,"QA":1,...}`, где `IMPL: 6` — первый свободный номер после существующих тикетов IMPL-001..IMPL-005 в archive/.

**Действие:** необходимо запустить стадию `allocate-ticket-ids` пайплайна, которая:
1. Рекурсивно просканирует `.workflow/tickets/` (включая `archive/`, `done/`, `review/`, `blocked/`)
2. Вызовет `get-next-id.js` для каждого префикса
3. Передаст `id_ranges_json` на вход декомпозиции

Без этого параметра создание тикетов невозможно — коллизия ID с архивными тикетами гарантирована by design.
