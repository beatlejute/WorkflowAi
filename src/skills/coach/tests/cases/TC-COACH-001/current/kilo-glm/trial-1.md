| F-3 | **MEDIUM** | Пробел в нумерации PLAN-002: IMPL-002..008 без IMPL-001 | Неизвестно — либо ранний архив без логирования, либо дефект нумерации при создании |

### Root Cause Analysis

**F-1 (Коллизия ID):** Физический автор новых тикетов — скил `decompose-plan`. Скрипт `pick-next-task.js` на шаге auto-correction архивировал тикеты PLAN-002 **до** запуска декомпозиции, но генератор ID внутри `decompose-plan` не имеет доступа к архивным тикетам или использует счётчик, сбрасываемый по активным тикетам. **Усиление нужно в скиле `decompose-plan`** (проверка на уникальность ID против всех состояний, включая archive), либо в инфраструктурном слое (глобальный счётчик / последовательность, не зависящая от статуса тикета).

**F-2 (status: default):** Декомпозиция завершилась корректно (5 тикетов созданы, выходной формат правильный), но результирующий статус неинформативен. Downstream стадии (`check-conditions`) не могут надёжно определить, была ли декомпозиция успешной или произошёл fallback.

### Рекомендации

1. **F-1:** Внести в `decompose-plan` обязательную проверку: ID нового тикета не должен совпадать ни с одним существующим ID в любом статусе (включая archive). Альтернативно — добавить префикс плана: `PLAN-003-IMPL-001`. Если генерация ID происходит в скрипте — добавить `grep` по `coach-backlog.yaml` и директории `tickets/` (включая archive) перед назначением номера.
2. **F-2:** Добавить в `decompose-plan` явную установку `status: success` при корректном завершении декомпозиции.
3. **F-3:** Проверить историю PLAN-002 на предмет IMPL-001 — существовал ли он и когда был архивирован.
