### Findings

#### F-1. Коллизия ID тикетов IMPL-002, IMPL-003 — **CRITICAL**

**Что:** Декомпозитор PLAN-003 создал тикеты с ID `IMPL-002` и `IMPL-003`, которые за 4 минуты до этого были архивированы как принадлежащие PLAN-002. Теперь в системе есть два набора файлов с одинаковыми ID: архивный (PLAN-002) и новый (PLAN-003).

**Root cause (первый порядок):** Скрипт `pick-next-task.js` (или `get-next-id.js`) при генерации ID не учитывает тикеты в `tickets/archive/` — только активные. Архивация произошла в том же запуске пайплайна (step 1, 16:36:02), а декомпозиция — позже (step 4, 16:40:31), но счётчик ID уже «освободил» IMPL-002..003.

**Root cause (второй порядок — скил):** Скил `decompose-plan` не содержит инструкции проверять уникальность ID относительно всех существующих файлов, включая `archive/`. Или скрипт `get-next-id.js` не сканирует `archive/` при поиске максимального ID.

**Последствия:**
- При поиске по ID (`grep`, `rg`, ссылки в тикетах) — неоднозначность: какой IMPL-002 имеется в виду?
- QA-001 зависит от IMPL-001 и IMPL-002 — если агент откроет архивный IMPL-002 вместо нового, выполнение пойдёт по неверному пути.
- FileGuard не сможет корректно отслеживать protected_files, если пути совпадают.

**Рекомендация:** `get-next-id.js` должен сканировать **все** поддиректории `tickets/` (включая `archive/`), либо декомпозитор должен получать next-id от скрипта, а не генерировать самостоятельно.

---

#### F-2. `status: default` вместо осмысленного статуса — **MEDIUM**

**Что:** Декомпозитор вернул `status: default`, хотя декомпозиция завершилась успешно (5 тикетов создано). Пайплайн перешёл на `check-conditions` по `status="default"`. Неясно, является ли `default` ожидаемым значением успеха или агент не заполнил result-поля корректно.

**Root cause:** Скил `decompose-plan` не задаёт жёсткий контракт на формат result (какой `status` означает успех, какой — ошибку). Агент вернул то, что счёт разумным.

**Рекомендация:** Добавить в скил `decompose-plan` обязательный контракт: `status: decomposed` при успехе, `status: error` при неудаче. Пайплайн должен реагировать на конкретные статусы, а не на `default`.

---

#### F-3. Пропущенные шаги 2–3 в логе — **LOW**

**Что:** Лог перепрыгивает с Step 1 (pick-first-task) на Step 4 (decompose-plan). Шаги 2 и 3 отсутствуют. Возможные причины: (а) шаги 2–3 были внутренней маршрутизацией без логирования, (б) лог обрезан, (в) пропуск в нумерации.

**Рекомендация:** Если шаги 2–3 — внутренняя маршрутизация (`GOTO`), добавить хотя бы DEBUG-логирование для отслеживания пути пайплайна. Это критично для диагностики коллизий.

---

#### F-4. Отсутствие валидации декомпозиции — **HIGH**

**Что:** После декомпозиции (step 4) пайплайн сразу перешёл к `check-conditions`, без промежуточной проверки: (а) что созданные тикеты не дублируют существующие/архивированные ID, (б) что граф зависимостей ацикличен, (в) что каждый тикет привязан к правильному `plan_id`.

**Root cause:** В пайплайне нет post-gate стадии после `decompose-plan`, которая валидировала бы результат декомпозиции перед продолжением.

**Рекомендация:** Добавить stage `validate-decomposition` между `decompose-plan` и `check-conditions`, который проверяет: уникальность ID (включая archive), привязку к plan_id, ацикличность графа зависимостей.
