### Слепая зона verify-artifacts (конкретный код)

`verify-artifacts.js:220-228` — `missing_files` проверяет только файлы из секции `### Изменённые файлы`. Агент при ghost execution:
1. Пишет любой текст в Result → `result_filled=true` ✓
2. Проставляет `[x]` в DoD → `dod_completion_pct=100` ✓
3. **Оставляет "Изменённые файлы" пустым** → `missing_files=[]` ✓

→ verify-artifacts возвращает `status: passed`, файлы физически не существуют.

### Что сделано (вторичная мера)

`execution-strategy.md` — добавлено в секцию «Призрачное выполнение»:
- **Признак**: «Секция «Изменённые файлы» пустая, хотя DoD требует создания или изменения файлов»
- **Условие #3**: DoD требует создания файла → "Изменённые файлы" обязательно заполнена
- **Чеклист**: явный пункт с пометкой «пайплайн проверяет механически»

Это страхует дисциплинированного агента, но **не решает проблему полностью.**

### Требуется от стейкхолдера: машинная защита

Попытки текстового усиления исчерпаны (3 инцидента, 3+ агентов). Нужен **IMPL-тикет**:

**Предмет**: Усилить `verify-artifacts.js` — добавить cross-check DoD → файлы на диске.

**Как**: Парсить DoD-пункты на паттерны упоминания файловых путей (`` `path/to/file` ``, `путь к файлу X`, `создать X`). Если найдены — проверить существование через `fs.existsSync()`. Дополнительный вариант: сравнивать mtime файлов ДО и ПОСЛЕ выполнения execute-task (runner-level).

**Почему только так**: агент, который игнорирует инструкцию «заполни "Изменённые файлы"», игнорирует и инструкцию «не делай ghost execution». Машинная проверка, не зависящая от дисциплины агента — единственная закрывающая мера для данного класса ошибок.
