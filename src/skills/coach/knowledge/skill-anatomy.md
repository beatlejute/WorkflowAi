# Эталонная анатомия скила

Справочник по структуре и содержанию идеального скила workflow-ai.

## Обязательная структура директорий

```
skill-name/
├── SKILL.md              # Ядро: роль, маршрутизация, принципы
├── README.md             # Документация для людей
├── workflows/            # Пошаговые алгоритмы по типам тикетов
├── knowledge/            # Экспертные знания домена
├── algorithms/           # Формализованные методы принятия решений
└── templates/            # Шаблоны вывода
```

## SKILL.md — обязательные секции

| Секция | Описание | Критерий качества |
|--------|----------|-------------------|
| **Frontmatter** | name, description, ticket_prefix | description — 2-3 предложения, prefix уникален |
| **Роль** | Кто агент, что делает / не делает | Конкретно, без абстракций |
| **Команда / RACI** | С кем взаимодействует (если есть) | Роли и зоны чёткие |
| **Маршрутизация** | Таблица: тип → триггеры → воркфлоу | Каждый тип → свой файл. **Только собственные типы скила** (по ticket_prefix), не project-wide типы из конфига |
| **Загрузка знаний** | Таблица: модуль → когда загружать | Все файлы knowledge/ перечислены |
| **Загрузка алгоритмов** | Таблица: алгоритм → когда загружать | Все файлы algorithms/ перечислены |
| **Шаблоны вывода** | Таблица: шаблон → когда использовать | Все файлы templates/ перечислены |
| **Принципы** | 3-7 принципов работы | Actionable, не декларативные |
| **Формат вывода** | Язык, стиль, конкретность | Специфичен для домена |
| **Границы компетенции** | Что НЕ входит в зону → куда направить | Перечислены все смежные зоны |

## ⛔ Принцип изоляции скилов

**Скилы НЕ должны знать о других скилах по имени.** В разных проектах набор скилов и ролей может быть другим.

| Что запрещено | Как правильно |
|---------------|--------------|
| Упоминание конкретных имён скилов/ролей (GML, PMA, CSM и т.д.) | «любой скил проекта», «соответствующий скил проекта», «скил-заказчик через тикет» |
| Примеры с конкретными ID ролей (GML-001, PMA-005) | Generic ID: XXX-001, example-skill |
| Таблица ролей в секции «Взаимодействие» | Универсальное описание: «Любой скил может создать тикет {PREFIX}-*» |
| Ссылки на инструменты конкретных скилов | «описаны в SKILL.md исполнителя» |

**Исключение:** скил может ссылаться на собственный ticket_prefix и на системные скилы, явно определённые в конфиге проекта (coach, execute-task).

### Изоляция от project-wide типов задач

**Скилы НЕ должны хардкодить типы задач из `config.yaml:task_types`.** Набор типов (`impl`, `fix`, `qa`, `admin` и т.д.) специфичен для проекта и управляется конфигом и pipeline.

| Что запрещено | Как правильно |
|---------------|--------------|
| Таблица маршрутизации по project-wide типам (`impl → execute.md`, `admin → admin.md`) | Маршрутизация по **собственным типам тикетов скила** (CREATE, AUDIT, ANALYZE) или единый универсальный воркфлоу |
| Knowledge-файл со списком всех типов задач проекта | Скил читает `type` из тикета и определяет подход по содержимому, а не по захардкоженной таблице |
| Отдельные воркфлоу под каждый project-wide тип | Воркфлоу по **этапам работы** (анализ, выполнение, верификация) или по **собственным типам скила** |

**Почему:** при добавлении нового типа в `config.yaml` скилы с хардкодом ломаются — неизвестный тип не попадает ни в один воркфлоу, агент импровизирует.

## Workflows — требования к качеству

- Каждый тип тикета из маршрутизации → свой файл
- Шаги пронумерованы, каждый шаг — конкретное действие
- Ссылки на knowledge/ и algorithms/ там, где нужны данные или метод
- Чеклисты или критерии завершения
- Типичные паттерны / примеры где уместно

## Knowledge — требования к качеству

- Данные структурированы: таблицы, списки, категории
- Маркер расширения: `<!-- РАСШИРЕНИЕ: добавляй ... ниже -->`
- Нет субъективных оценок без данных или источников
- Формулы и пороги вынесены явно
- Актуальность: дата последнего обновления или источник

## Algorithms — требования к качеству

- Чёткий вход (что подаём) и выход (что получаем)
- Пошаговый алгоритм с конкретными действиями
- Пороги, формулы, весовые коэффициенты — числами
- Пример применения
- Граничные случаи / исключения

## Templates — требования к качеству

- Все поля обязательны или помечены `(опционально)`
- Placeholder-подсказки в каждом поле
- Формат markdown, единообразный стиль
- Покрытие: для каждого типа результата — свой шаблон

## README.md — обязательные секции

- Название и краткое описание
- Структура (дерево файлов)
- Как это работает (flow)
- Как расширять (по каждому типу компонента)
- Примеры расширений

## Shared Knowledge — межскиловые знания

Когда knowledge-модуль нужен 2+ скилам — он выносится в директорию `shared/` на одном уровне со скилами:

```
skills/
├── shared/               # Проектно-специфичные знания, общие для нескольких скилов
│   ├── README.md         # Индекс: таблица модулей с описанием и списком потребителей
│   ├── product-ui-map.md # Пример: карта UI тестируемого продукта
│   └── ...
├── skill-a/
├── skill-b/
└── ...
```

### README.md — обязательный индекс

```markdown
# Shared Knowledge Modules

| Модуль | Описание | Используется в |
|--------|----------|----------------|
| `module-name.md` | Что содержит | skill-a, skill-b |
```

### Правила

| Правило | Описание |
|---------|----------|
| **Критерий выноса** | Контент используется 2+ скилами с пересечением 70%+ |
| **Скилы — универсальные, shared — проектный** | Скилы переносятся между проектами as-is. Shared содержит проектно-специфичные знания (карта UI продукта, предметная область, API-контракты) |
| **Ссылка из скила — через glob** | В таблице «Загрузка знаний» скила: `../shared/*` → «проверь индекс, загрузи релевантные». Не хардкодить имена конкретных модулей |
| **Синхронность обновлений** | Изменение в shared-модуле автоматически доступно всем потребителям |
| **Domain-контекст — в скиле** | Если скилу нужна специфичная интерпретация shared-знаний, она остаётся в его собственных knowledge/workflows (thin wrapper) |

### Антипаттерны

| Антипаттерн | Почему плохо | Как правильно |
|-------------|-------------|---------------|
| Дублировать одни и те же данные в knowledge/ нескольких скилов | Рассинхрон при обновлении | Вынести в shared/ |
| Хардкодить `../shared/product-ui-map.md` в SKILL.md | Скил перестаёт быть универсальным | `../shared/*` → проверить README |
| Класть в shared/ knowledge, нужное одному скилу | Засорение, ложная связность | Оставить в knowledge/ самого скила |
| Делать shared/ вложенным в конкретный скил | Нарушает семантику: shared принадлежит всем | shared/ — на одном уровне со скилами |
