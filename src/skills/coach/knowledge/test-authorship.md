# Авторство тест-кейсов регрессионных тестов скилов

Руководство для коуча по написанию тест-кейсов при работе с регрессионными тестами скилов.

---

## 1. Дерево выбора слоя (L0 / L1 / L2)

Выбери слой по первому подходящему условию:

| Слой | Название | Когда использовать |
|------|----------|-------------------|
| **L0** | static | **Всегда** — если нужно проверить, что агент записал конкретный инвариант в файл (наличие раздела, запрета, поля). Не зависит от модели. |
| **L1** | deterministic | Когда у output есть наблюдаемая сигнатура: структура YAML/JSON, наличие обязательных ключей, формат ID. Подходит, если ответ детерминирован при заданном входе. |
| **L2** | rubric | Когда нужно проверить **принцип рассуждения** агента: правильно ли выбрана root cause, корректно ли применён принцип. Оценивается поведение, а не текст. |

Правило: если сомневаешься между L0 и L1 — выбирай L0. L2 только тогда, когда L0/L1 физически не применимы.

---

## 2. Правила написания static-anchor

**Семантический инвариант, не дословный текст:**

- Anchor должен выражать *правило*, нарушение которого тест должен поймать.
- Anchor **отвергает** любое поведение, нарушающее правило, а не просто проверяет «строка существует в файле».
- Один anchor — одно правило. Несколько правил → несколько anchor'ов.

**Пример плохого anchor:** `"git commit"` — существует в любом упоминании темы.

**Пример хорошего anchor:** `"коуч не выполняет git commit ни при каких условиях"` — семантически проверяет запрет, а не присутствие слова.

---

## 3. Минимизация фикстуры

- Фикстура — реальный артефакт ±20 строк вокруг инцидента.
- **Не включай** в фикстуру весь лог или полный тикет.
- Критерий минимума: убери строку → тест перестаёт проверять нужное поведение? Если нет — строка лишняя.
- Для log-фикстур: вырезай только те шаги, где произошёл инцидент + 5 строк контекста до и после.

---

## 4. Формулировка rubric-критерия

Rubric проверяет **процесс рассуждения**, а не конкретный текст ответа:

- Формулируй как: «Агент должен [определить / выбрать / проверить] [что именно] [по какому признаку]».
- Не указывай конкретные слова, которые агент должен произнести — указывай логический шаг.
- Rubric должен оставаться валидным при переформулировке инструкций скила.

**Пример плохого rubric:** «Ответ содержит слово "root cause"».

**Пример хорошего rubric:** «Агент идентифицировал физический скил-источник артефакта, а не предметный владелец».

## 5. Выбор `target_agents[]` на уровне скила

Заполняется автором-коучем вручную при первой настройке, сверяясь с `pipeline.yaml` проекта.

Перечислить как минимум все модели, на которых скил реально гоняется в пайплайне:
- основная модель скила,
- fallback-модель (если задана),
- модели из секций `agent_by_type` и `agent_by_attempt`.

**Дополнительные модели для сравнения разрешены.** `target_agents[]` может включать модели сверх pipeline-набора — они выступают как «контрольная группа» для выявления регрессий формулировок скила на более широкой выборке. Такие модели помечаются комментарием (см. пример в `coach/tests/index.yaml`). Наличие «лишних» моделей — не ошибка и не повод сокращать список.

Заполняется один раз в `tests/index.yaml` и наследуется всеми тест-кейсами скила. Не дублировать в каждом кейсе отдельно.

**⛔ Сокращать `target_agents[]` (на уровне `index.yaml` или через per-case override) коуч не имеет права без явного согласия стейкхолдера.** Это относится к любому удалению модели из списка, включая случай «модель не используется в pipeline.yaml». Обоснование: `target_agents[]` — это заявленная сфера регрессии скила, а не техническая конфигурация; сокращение сужает покрытие тестов. См. подробный запрет ниже в секции «Override на уровне кейса».

**Сверка с pipeline.yaml:**

Runner автоматически проверяет каждый agent-id из `target_agents[]` против списка `agents` в `pipeline.yaml`. Если агент не найден — fail-fast с сообщением:

```
Agent 'invalid-agent-id' from target_agents[] not found in pipeline.yaml → agents[]
```

Порядок резолва `pipeline.yaml`:
1. Приоритет: `<project_root>/.workflow/config/pipeline.yaml` (через `find-root.mjs`)
2. Fallback: `<package_root>/configs/pipeline.yaml`

Путь к используемому файлу логируется при старте runner'а.

**Override на уровне кейса:**

Точечный override допустим — если конкретный тест-кейс требует другую модель, укажи `execution.target_agents[]` в файле кейса. Это переопределит значение из `index.yaml` для данного кейса.

**⛔ Override target_agents — это решение стейкхолдера, не коуча.** Убирать модель из списка или сужать его — **изменение сферы тестирования**, не техническая правка. Коуч **не имеет права** проактивно применять per-case override target_agents при провале модели на тесте, даже если:
- модель выдаёт нестабильные/обрезанные ответы;
- есть прецедент в бэклоге, где похожий override уже применялся;
- формально «модель физически не может пройти тест».

Все эти случаи — повод **явно согласовать** с стейкхолдером в отдельном сообщении, показав: (а) диагностику, что именно модель не может сделать, (б) три варианта действий (оставить как есть с honest baseline / исследовать глубже / исключить), (в) дождаться выбора. Применение override без явного согласия — перекладывание решения стейкхолдера на коуча и нарушение **границ компетенции** (см. SKILL.md, раздел «Границы компетенции»).

**Антипаттерн:** «в прецеденте X модель была исключена, применяю тот же паттерн молча». Прецедент — это не карт-бланш. Проверь: стейкхолдер явно запросил исключение в прецеденте или коуч предложил и получил согласие? Если второе — всегда повторяй согласование для нового случая, даже если кажется аналогичным.

**CLI-флаги для запуска:**

- `--agent <id>` — прогон только на конкретной модели из `target_agents[]`
- `--primary-only` — только первая модель из `target_agents[]` (без fallback)

## 6. Сравнение прогонов через git

Для отслеживания регрессий между прогонами используй стандартные git-инструменты:

| Задача | Команда |
|--------|---------|
| Посмотреть изменения в мета-файле | `git diff current/meta.json` |
| История прогонов | `git log current/meta.json` |
| Кто изменил конкретную строку | `git blame current/meta.json` |

Никакой специальной инфраструктуры не требуется — `current/meta.json` версионируется как обычный файл.

---

## 7. Кто делает commit

**⛔ Коуч не выполняет `git add`, `git commit`, `git push`, `git checkout`, `git reset` ни при каких условиях.**

Коммит `current/` делает **исключительно пользователь**.

Роль коуча после прогона тестов:
1. Сообщить verdict runner'а (pass / fail / error).
2. Перечислить затронутые файлы.
3. Остановиться — дальнейшие git-операции выполняет пользователь.

## 8. Антипаттерны авторства

| Антипаттерн | Почему плохо | Как правильно |
|-------------|-------------|---------------|
| Тестировать дословную формулировку инструкции | Хрупко: переформулировка инструкции ломает тест, не меняя поведение | Тестировать семантический инвариант правила |
| Over-specifying expected output | Привязка к конкретным словам делает тест зависимым от стиля модели | Проверять принцип, а не точный текст |
| Фикстура = полный лог/тикет | Тест медленный, нечитаемый, хрупкий к несвязанным изменениям | Минимальная фикстура ±20 строк вокруг инцидента |
| L2 вместо L0 для проверки файла | L2 дороже и недетерминирован; если можно проверить файл — используй L0 | Выбирай наименьший подходящий слой |
