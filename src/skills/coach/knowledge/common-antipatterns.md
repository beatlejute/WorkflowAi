# Типичные антипаттерны в скилах

Чеклист проблем при аудите и ревью скилов.

## Структурные (S)

| # | Антипаттерн | Как исправить |
|---|------------|---------------|
| S1 | Осиротевшие файлы (не в SKILL.md) | Добавить в таблицу загрузки или удалить |
| S2 | Битые ссылки | Создать файл или исправить ссылку |
| S3 | Пустые директории | Заполнить или убрать |
| S4 | Монолитный SKILL.md | Декомпозировать в воркфлоу |
| S5 | Нет README | Создать по эталону |

## Содержание (C)

| # | Антипаттерн | Как исправить |
|---|------------|---------------|
| C1 | Абстрактная роль | Описать роль, границы, зону ответственности |
| C2 | Размытые принципы | Что делать, когда, как измерить |
| C3 | Воркфлоу-декларация без шагов | Конкретные шаги, чеклисты, пороги |
| C4 | Knowledge без структуры | Таблицы, заголовки, маркеры расширения |
| C5 | Algorithm без формул | Числовые критерии, формулы, пороги |
| C6 | Template без подсказок | Примеры и placeholder в каждое поле |

## Поведение (B)

| # | Антипаттерн | Как исправить |
|---|------------|---------------|
| B1 | Выход за границы | Уточнить «Ты НЕ делаешь» |
| B2 | Дублирование знаний | Вынести в один модуль, ссылаться |
| B3 | Нет маршрута для типа | Добавить воркфлоу/маршрутизацию |
| B4 | Игнорирование knowledge | Добавить ссылки в шагах воркфлоу |
| B5 | Нет fallback | Правило «Если тип не определяется — ...» |

## Изоляция (I)

| # | Антипаттерн | Как исправить |
|---|------------|---------------|
| I1 | Хардкод имён ролей/скилов | → «любой скил проекта», generic ID (XXX-NNN) |
| I2 | Примеры с конкретными ID ролей | → XXX-001, example-skill |
| I3 | Таблица ролей во Взаимодействии | → «Любой скил может создать тикет {PREFIX}-*» |
| I4 | Ссылки на инструменты других скилов | → «описаны в SKILL.md исполнителя» |
| I5 | Границы компетенции с именами | → «соответствующий скил проекта» |

**Проверка:** `grep -rE "GML|PMA|CSM|CRO|PMK" src/skills/` → 0 результатов.

## Изоляция от конфигурации (IC)

| # | Антипаттерн | Как исправить |
|---|------------|---------------|
| IC1 | Хардкод project-wide типов задач | Скил определяет подход по содержимому тикета |
| IC2 | Knowledge-дубликат конфига | Удалить, источник правды — конфиг |
| IC3 | Воркфлоу под каждый project-wide тип | Организовать по этапам работы или собственным типам |

## Scope management (SC)

| # | Антипаттерн | Как исправить |
|---|------------|---------------|
| SC1 | Scope creep через DoD | Запретить execute-task создавать тикеты |
| SC2 | Декомпозиция без scope-guard | Проверка «Включено/Исключено из scope» |
| SC3 | Тикеты-сироты без parent_plan | Обязательно заполнять parent_plan |
| SC4 | Каскадная декомпозиция | Только decompose-plan/gaps создают тикеты |
| SC5 | Горизонт за пределами плана | Future-задачи → рекомендации, не тикеты |
| SC6 | Каскад execute→plan→decompose | 3 уровня защиты (см. execute-task SKILL.md) |
| SC7 | Автодекомпозиция без human-approval | Только планы со `status: approved` |

## Масштабирование (M)

| # | Антипаттерн | Как исправить |
|---|------------|---------------|
| M1 | Нет маркеров расширения | `<!-- РАСШИРЕНИЕ: ... -->` в конец |
| M2 | Жёсткая привязка | Параметризовать или вынести в конфиг |
| M3 | Нет инструкций по расширению | Секция «Как расширять» в README |
