# check-relevance

Скил проверки актуальности тикета перед выполнением. Гейт между перемещением в `in-progress` и передачей скилу выполнения.

## Структура

```
check-relevance/
├── SKILL.md                          # Ядро: роль, принципы, таблицы загрузки
├── README.md                         # Документация (этот файл)
├── knowledge/
│   ├── ticket-states.md              # Состояния тикетов, критерии неактуальности
│   └── plan-states.md                # Состояния планов и их семантика
└── algorithms/
    └── relevance-check.md            # Формализованный алгоритм проверки
```

## Как это работает

1. Тикет перемещается в `in-progress/`
2. Скил загружает алгоритм `algorithms/relevance-check.md`
3. Последовательно проверяет: ревью → блокировки → план → DoD → зависимости
4. Первая сработавшая проверка → `irrelevant` (short-circuit)
5. Все проверки пройдены → `relevant`
6. При `irrelevant` — добавляет запись в таблицу ревью тикета

**Принцип fail-safe:** при любой ошибке или неопределённости → `relevant`.

## Как расширять

### Добавить новый knowledge-модуль

1. Создай файл в `knowledge/` с маркером `<!-- РАСШИРЕНИЕ: ... -->`
2. Добавь запись в таблицу «Загрузка знаний» в `SKILL.md`

### Добавить новый критерий проверки

1. Добавь шаг в `algorithms/relevance-check.md` (в правильное место по приоритету)
2. Добавь строку в «Таблицу решений»
3. Добавь значение `reason` в `SKILL.md` → «Формат вывода»
4. При необходимости добавь данные в соответствующий knowledge-модуль

### Добавить новое состояние плана

1. Добавь строку в таблицу в `knowledge/plan-states.md` после маркера расширения
2. Обнови шаг проверки плана в `algorithms/relevance-check.md` если нужно

## Примеры расширений

**Новый критерий: «тикет просрочен»**
- `knowledge/ticket-states.md` — добавить критерий `deadline_expired`
- `algorithms/relevance-check.md` — добавить шаг между 2 и 3: проверка `deadline` в frontmatter
- `SKILL.md` — добавить `deadline_expired` в допустимые значения reason
